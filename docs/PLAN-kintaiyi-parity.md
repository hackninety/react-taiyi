## 三、实施方案

**后端（python-taiyi，1 个提交）**
1. `POST /api/taiyi/pan`：`t.pan(ji, acum, enable_game_theory)` → JSON 安全化（tuple/键统一 str）后透传；这是 A 表全部数据的唯一入口。
2. `GET /api/taiyi/life`：命法卷二十扩展（kintaiyi mingfa 相关方法群，逐一核对方法名后透传）。
3. `GET /api/docs/{name}`：白名单 example/disaster/guji/instruction/tutorial/update，读上游包内 `docs/*.md` 原文返回（Cache-Control 1d）。
4. 冒烟：pan JSON 可序列化、全键非空抽查；bump 部署。

**前端（react-taiyi，按 P0→P3 分批提交）**
- **P0 解释总汇**：`fetchPan()` + 「解釋」抽屉卡（分组折叠展示 A 表断法/所主术/值宿断事/釋格局全文）；数据仅远程（本地源时提示"解释内容由 kintaiyi 后端提供"）。
- **P1 統運与卦象**（用户点名重点）：統運入卦卡（时间轴 React 重写 yun_timeline）、統運查詢（任意年）、統運歷史對照；卦象總述/動爻辭/六爻觀象/變卦納甲卡；流年月日時分卦運横向时间轴（重写 hex_timeline，滑动查未来）。
- **P2 诸卷专题卡**：军事（五陣八陣 SVG 置旗图）、分野、軌運、州國災變、十精雲氣、太乙/文昌九星、五運六氣/五音、金函玉鏡、博弈论矩阵。
- **P3 独立页**：局數史例（交互时间轴+即时排盘）、災異統計、古籍書目、看盤要領、更新日誌；导航从两个 tab 扩为下拉/多 tab。
- 每批同步：AI 导出把新增内容并入 meta/analysisContext（流派口径标注规则不变）、说明页、README、必要的守卫测试（pan 键存在性 + docs 端点 200）。

**规模预估**：后端半天；P0+P1 约一次长会话；P2、P3 各一次会话。全程无算法移植，漂移风险=0（数据直出上游）。

## 四、注意事项

- `pan()` 单次调用较重（内部重复算多次），后端可加 60s LRU 缓存（同参数）；不要在前端逐卡片单独请求。
- pan 返回含 tuple/中文键/嵌套 dict——序列化层统一转 list/str，键保持中文（前端按键名直读，与上游文档一致）。
- 流分计（ji=4）部分卷内容上游可能不适用/抛错——逐计式冒烟，异常键置 null 并在 UI 隐藏。
- 命法卷二十方法名需先在 `src/kintaiyi/kintaiyi.py` 核对（ming_* / *_xingxian / zhao_you 等），勿凭 label 猜。
- 看盤要領/教程 md 为繁体原文——直接原文展示并标注出处，不做简繁转换（与卦名繁体口径一致）。
